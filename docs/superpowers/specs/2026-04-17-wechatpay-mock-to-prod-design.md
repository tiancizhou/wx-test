# 微信支付 mock 到正式上线改造设计

## 背景

当前项目已具备商户模式下的微信支付基础骨架：

- 后端已有 JSAPI 下单、支付回调、查单、关单、退款、退款回调接口
- 前端已有公众号内 `WeixinJSBridge.invoke('getBrandWCPayRequest')` 调起逻辑
- 系统通过 `WX_PAY_MOCK` 支持开发期 mock 联调

但当前实现还没有达到“配置补齐、关闭 mock 后即可直接正式上线”的标准，原因包括：

1. 支付回调与退款回调未实现生产级验签
2. 支付回调未校验金额、`appid`、`mchid`
3. mock 模式直接在建单接口内把订单改为已支付，未与正式支付成功链路统一
4. 正式模式下缺少关键支付配置时，系统不会在启动阶段硬失败

本设计目标是在**保留 mock 联调能力**的前提下，将商户模式支付链路补齐到正式上线标准，使后续只需补齐证书与配置并关闭 mock，即可切换到真实支付。

---

## 目标

本次改造的目标为：

1. 保留 `WX_PAY_MOCK=true` 时的联调能力
2. 将 mock 支付从“直接写死支付成功”改造成“模拟支付成功后走统一业务落库链路”
3. 为商户模式补齐正式支付的必需安全能力：
   - 支付回调验签
   - 退款回调验签
   - 支付回调金额、`appid`、`mchid` 校验
4. 在 `WX_PAY_MOCK=false` 时强制检查关键支付配置，缺失则启动失败
5. 保持现有前后端路由与页面结构基本稳定，避免超出本次最小可上线标准的重构

---

## 非目标

本次改造**不包含**以下内容：

1. 账单下载与自动对账能力
2. 定时主动查单兜底机制
3. mock 模式下的取消支付、超时未支付等复杂分支模拟
4. 多支付方式扩展（H5、Native、APP、小程序等）
5. 合单支付、分账、服务商模式适配
6. 大规模重构为独立 provider 抽象层

本次仅实现“mock 可联调 + 正式模式可安全上线”的最小闭环。

---

## 设计原则

### 1. 订单状态推进逻辑只保留一套

无论是正式微信支付回调，还是 mock 模式的模拟支付成功，都必须汇入统一的支付成功处理逻辑。不得再保留“mock 直接改订单状态、正式模式走另一套逻辑”的分叉。

### 2. 回调按生产标准执行

支付回调与退款回调必须遵循：

1. 读取头部与请求体
2. 先验签
3. 再解密
4. 再做业务字段校验
5. 最后幂等更新订单状态

### 3. 正式模式必须是硬约束

当 `WX_PAY_MOCK=false` 时，若关键支付配置不完整，应用应在启动阶段直接失败，而不是允许服务处于“能启动但支付一定报错”的半可用状态。

### 4. 范围最小化

在达成上线标准前提下，尽量复用现有 `main.py` 路由与前端调用结构，只做必要重组，不引入超出需求的抽象层。

---

## 当前问题总结

### 问题 1：mock 与正式链路不等价

当前 `POST /pay/create` 在 mock 模式下直接将订单状态从 `UNPAID` 改为 `ORDERED`，并直接增加销量。这导致 mock 联调绕过了正式支付中“创建待支付订单 → 支付成功 → 回调推进状态”的核心阶段。

### 问题 2：支付回调未做正式验签

当前支付回调只做了解密尝试，没有使用微信支付平台公钥校验 `Wechatpay-Signature`，不符合正式商户模式上线要求。

### 问题 3：退款回调未做正式验签

退款回调同样只做解密，没有实现正式验签。

### 问题 4：支付回调业务校验不足

当前支付回调没有校验：

- 回调金额是否与本地订单金额一致
- 回调 `appid` 是否匹配当前应用
- 回调 `mchid` 是否匹配当前商户号

### 问题 5：正式支付配置约束不够严格

当前即使 `WX_PAY_MOCK=false`，系统也未在启动阶段统一校验支付正式配置是否完整。

---

## 目标结构

本次改造后，支付链路应形成：

- 一套统一的订单状态推进逻辑
- 两种支付来源：
  - 正式微信支付回调
  - mock 模式模拟支付成功
- 一套统一的正式配置校验机制

后端职责分层如下：

1. **路由层**：负责接收请求、提取参数、做模式分流
2. **支付验证层**：负责验签、解密、提取微信支付回调数据
3. **支付应用层**：负责支付成功/退款成功的业务校验与幂等落库
4. **配置校验层**：负责在应用启动时校验正式模式所需配置

---

## 后端改造设计

### 一、统一支付成功处理函数

新增内部函数 `apply_payment_success(...)`，职责如下：

输入包含：

- `order_id`
- `transaction_id`
- `paid_amount`
- `appid`
- `mchid`
- 数据库会话

处理逻辑：

1. 根据 `order_id` 查询订单及商品信息
2. 订单不存在则报错
3. 若订单已不是 `UNPAID`，按幂等成功处理直接返回
4. 校验 `paid_amount == order.total_fee`
5. 校验 `appid == settings.APP_ID`
6. 校验 `mchid == settings.MCH_ID`
7. 将订单状态更新为 `ORDERED`
8. 写入 `transaction_id`
9. 增加对应商品销量
10. 提交事务

该函数是正式回调和 mock 成功链路共享的唯一支付成功入口。

### 二、统一退款成功处理函数

新增内部函数 `apply_refund_success(...)`，职责如下：

输入包含：

- `order_id`
- `refund_id`
- 数据库会话

处理逻辑：

1. 查询订单及商品信息
2. 订单不存在则按失败返回
3. 若订单已是 `REFUNDED`，按幂等成功处理直接返回
4. 将订单状态更新为 `REFUNDED`
5. 更新 `refund_id`
6. 回退商品销量（最低不小于 0）
7. 提交事务

该函数作为退款回调的统一落库入口。

### 三、调整 `/pay/create`

保留现有接口路径 `POST /pay/create`，但重构行为：

共同逻辑：

1. 校验商品有效性
2. 根据商品价格与数量计算 `total_fee`
3. 创建 `UNPAID` 订单并落库

正式模式（`WX_PAY_MOCK=false`）：

1. 调用微信支付统一下单
2. 返回真实 `pay_params`
3. 订单继续保留 `UNPAID`，等待支付回调推进状态

mock 模式（`WX_PAY_MOCK=true`）：

1. 不直接修改订单状态
2. 返回：
   - `mock: true`
   - `order_id`
   - `status: UNPAID` 或等价语义
3. 由前端在确认 mock 支付成功时调用专用 mock 确认接口

### 四、新增 mock 支付确认接口

新增接口：`POST /pay/mock/confirm/{order_id}`

职责：

1. 仅在 `WX_PAY_MOCK=true` 时允许调用
2. 校验当前用户对订单有访问权限
3. 构造一份受控的 mock 支付结果：
   - `transaction_id` 可生成固定前缀的 mock 值，如 `mock_{order_id}`
   - `paid_amount = order.total_fee`
   - `appid = settings.APP_ID`
   - `mchid = settings.MCH_ID`
4. 调用 `apply_payment_success(...)`
5. 返回支付成功结果

本接口只用于开发联调，不进入正式模式。

### 五、支付回调重构

保留接口：`POST /pay/notify`

处理逻辑调整为：

1. 读取请求头：
   - `Wechatpay-Timestamp`
   - `Wechatpay-Nonce`
   - `Wechatpay-Signature`
   - `Wechatpay-Serial`
2. 读取 body
3. 调用正式验签函数校验签名
4. 使用 `API_V3_KEY` 解密 `resource`
5. 校验：
   - `event_type == TRANSACTION.SUCCESS`
   - `resource.trade_state == SUCCESS`
   - `resource.out_trade_no` 存在
6. 取出：
   - `out_trade_no`
   - `transaction_id`
   - `amount.total`
   - `appid`
   - `mchid`
7. 调用 `apply_payment_success(...)`
8. 返回 HTTP 200

若验签失败、解密失败、金额不一致、商户号不一致、AppID 不一致，应返回失败响应，不落库。

### 六、退款回调重构

保留接口：`POST /refund/notify`

处理逻辑调整为：

1. 读取请求头
2. 读取 body
3. 调用正式验签函数校验签名
4. 解密回调资源
5. 校验 `refund_status == SUCCESS`
6. 取出 `out_trade_no` 与 `refund_id`
7. 调用 `apply_refund_success(...)`
8. 返回 HTTP 200

### 七、查单、关单、退款接口

现有查单、关单、退款申请接口继续保留，只做必要一致性修正：

- mock 模式下继续允许本地联调
- 正式模式下保留真实 API 调用
- 不额外扩大能力范围

其中退款申请仍使用固定 `out_refund_no = R{order_id}`，这样在同一订单单次退款场景下可避免误生成多个退款单号；由于本次不支持部分退款多次场景，该策略可接受。

## 微信支付验签设计

### 一、正式验签要求

需要新增正式回调验签能力，逻辑包括：

1. 根据回调头取出：
   - 时间戳
   - 随机串
   - 签名值
   - 微信支付平台证书序列号
2. 构造验签串：

```text
时间戳\n
随机串\n
响应体\n
```

实际为：

```text
{timestamp}\n{nonce}\n{body}\n
```

3. 使用微信支付平台公钥验证签名
4. 拒绝超出合理时间窗口的回调请求

### 二、公钥来源

本次按“最小可上线标准”处理，建议通过环境变量或文件路径提供微信支付平台公钥，避免在本次范围内继续扩展“自动下载并轮换平台证书”能力。

新增正式模式依赖配置之一：

- `WX_PAY_PLATFORM_PUBLIC_KEY_PATH`

如需兼容证书序列号校验，可再增加：

- `WX_PAY_PLATFORM_SERIAL`

本次推荐最小实现为：

- 启动时加载平台公钥
- 回调验签时使用该公钥验证
- 若配置了平台序列号，则要求 `Wechatpay-Serial` 匹配

### 三、mock 与验签关系

mock 支付确认接口不伪造微信签名，不模拟完整加密报文。它只调用统一业务落库函数，不进入正式验签流程。

这样既避免 mock 复杂化，也保证正式回调逻辑不会被 mock 逻辑污染。

## 配置校验设计

### 一、正式模式必需配置

当 `WX_PAY_MOCK=false` 时，应用启动前必须校验以下配置全部存在：

- `WX_MCH_ID`
- `WX_MCH_SERIAL_NO`
- `WX_MCH_PRIVATE_KEY_PATH`
- `WX_API_V3_KEY`
- `WX_PAY_PLATFORM_PUBLIC_KEY_PATH`

可选但推荐：

- `WX_PAY_PLATFORM_SERIAL`

### 二、校验时机

推荐在应用启动阶段执行统一校验，例如：

- `lifespan` 初始化期间
- 或配置模块导入时的显式校验函数

要求：

- `PAY_MOCK=true` 时，允许正式支付配置缺失
- `PAY_MOCK=false` 时，缺任一必需配置即抛出异常并阻止启动

### 三、生产配置约束

与支付正式链路相关的敏感配置不得依赖不安全默认值。对于正式支付关键字段，应明确要求通过环境变量注入。

本次至少应保证：

- 不因缺少支付关键配置而进入“表面启动成功、实际支付全失败”的状态
- 关键支付私钥仍通过文件读取，不硬编码到代码中

## 前端改造设计

### 一、客户下单流程调整

当前前端客户页面在 mock 模式下收到 `/pay/create` 返回 `mock=true` 后直接提示“下单成功”。本次需调整为：

1. 调用 `/pay/create`
2. 若返回 `mock=true`：
   - 再调用 `/pay/mock/confirm/{order_id}`
   - 成功后提示“下单成功”
3. 若返回真实 `pay_params`：
   - 继续调用 `WeixinJSBridge.invoke`

### 二、前端目标

前端不需要模拟微信支付 UI，也不新增复杂 mock 面板。本次只保证 mock 下能触发“模拟支付成功”，从而让订单进入与正式回调一致的成功落库路径。

---

## 幂等与状态规则

### 一、支付成功幂等

`apply_payment_success(...)` 必须满足：

- 订单当前为 `UNPAID`：执行正式更新
- 订单当前已不是 `UNPAID`：视为重复支付成功通知，直接按幂等成功返回

不得重复：

- 修改订单状态
- 覆盖错误交易号
- 增加商品销量

### 二、退款成功幂等

`apply_refund_success(...)` 必须满足：

- 订单当前不是 `REFUNDED`：执行退款落库
- 订单当前已是 `REFUNDED`：按幂等成功返回

不得重复：

- 回退销量
- 改写异常状态

### 三、订单状态边界

本次支付成功只允许把订单从 `UNPAID` 推进到 `ORDERED`。

本次退款成功只允许把订单推进到 `REFUNDED`。

不扩展其它状态机优化。

---

## 错误处理原则

### 一、支付回调

以下情况必须拒绝处理并返回失败响应：

- 验签失败
- 解密失败
- 缺少订单号
- 回调金额与本地订单金额不一致
- 回调 `appid` 与本地配置不一致
- 回调 `mchid` 与本地配置不一致

### 二、mock 确认接口

以下情况返回错误：

- `PAY_MOCK=false` 时访问 mock 接口
- 订单不存在或无权限访问
- 订单状态不是 `UNPAID`

### 三、正式模式配置不完整

在应用启动阶段直接失败，不允许进入运行态。

---

## 验收标准

改造完成后，必须满足以下验收标准：

### 1. mock 联调能力

- `WX_PAY_MOCK=true` 时，客户下单后不再在 `/pay/create` 里直接把订单改为已支付
- 前端通过 mock 确认接口推进订单到 `ORDERED`
- mock 支付成功最终调用统一的支付成功处理函数

### 2. 正式支付安全能力

- 支付回调实现真实验签
- 退款回调实现真实验签
- 支付回调处理前校验金额、`appid`、`mchid`

### 3. 正式模式配置约束

- `WX_PAY_MOCK=false` 且缺少关键支付配置时，应用启动失败

### 4. 幂等性

- 重复支付通知不会重复增加销量
- 重复退款通知不会重复回退销量

### 5. 切换成本

- 未来证书、商户号、APIv3 Key、平台公钥等配置到位后，只需：
  - 配置正式环境变量
  - 设置 `WX_PAY_MOCK=false`
- 无需再改支付业务代码，即可切换为真实商户模式支付

---

## 实施范围提示

本设计适合作为一次独立的最小支付上线改造。它聚焦于：

- mock 与正式链路统一
- 回调安全补齐
- 配置强约束

不会扩展到账单、对账、复杂失败场景模拟或支付架构全面重构，因此范围清晰，适合直接进入实现计划。
