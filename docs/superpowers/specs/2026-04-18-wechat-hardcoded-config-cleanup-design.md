# 微信配置硬编码测试值清理设计

## 背景

当前仓库里仍存在与微信配置相关的真实测试默认值与前端硬编码：

- 后端 `backend/wechat/config.py` 仍为 `WX_TOKEN`、`WX_APP_ID`、`WX_APP_SECRET` 提供真实测试默认值
- 前端 `frontend/index.html` 仍直接写死测试 `WX_APP_ID`
- 前端 `frontend/customer.html` 与 `frontend/merchant.html` 虽然会请求 `/config/frontend`，但初始化阶段仍先写死测试 `WX_APP_ID`

这会带来两个问题：

1. 本地或线上环境即使没有正确配置，也可能因为默认值而“看起来能跑”，掩盖配置缺失
2. 测试号配置被硬编码在代码中，不利于后续正式上线与仓库安全管理

本次设计目标是在**尽量小的改动范围**内去掉这些硬编码测试值，让前端只从后端配置接口读取 `app_id`，并在缺少必要配置时给出明确失败，而不是静默回退到测试值。

---

## 目标

本次改造的目标为：

1. 后端不再为 `WX_TOKEN`、`WX_APP_ID`、`WX_APP_SECRET` 提供真实测试默认值
2. 前端页面不再写死测试 `WX_APP_ID`
3. 前端只从后端 `/config/frontend` 获取 `app_id`
4. 当 `WX_APP_ID` 缺失时，前后端都应表现为**明确失败**，不再回退到测试值
5. 保持现有页面、接口与登录流程结构基本不变，优先保证现有功能可用

---

## 非目标

本次改造**不包含**以下内容：

1. 不重构现有 OAuth 登录流程
2. 不新增复杂配置中心或前端构建流程
3. 不调整 `BASE_URL` 的现有来源与部署方案
4. 不引入新的环境文件加载框架
5. 不扩大到支付证书、支付商户配置或其他与本次硬编码清理无关的改造

---

## 设计原则

### 1. 缺配置时显式失败

本次改造采用严格模式：缺少必要微信配置时，应直接返回明确错误，而不是回退到仓库中的测试值。

### 2. 前端配置来源唯一

前端页面所需 `app_id` 统一只从后端 `/config/frontend` 获取，不再存在“页面先写死测试值、再被接口覆盖”的双来源模式。

### 3. 范围最小化

优先只修改与当前问题直接相关的配置定义、配置接口和 3 个前端页面初始化逻辑，不做额外抽象。

### 4. 保持现有可用路径

只要环境里已正确提供 `WX_APP_ID`，现有首页、顾客页、商家页的 OAuth 跳转行为应保持不变。

---

## 当前问题总结

### 问题 1：后端配置存在真实测试默认值

`backend/wechat/config.py` 当前写法使得以下环境变量即使未设置，也会自动落到真实测试号配置：

- `WX_TOKEN`
- `WX_APP_ID`
- `WX_APP_SECRET`

这会掩盖配置遗漏，也可能把测试配置随代码传播到不该存在的环境中。

### 问题 2：前端首页直接写死测试 `WX_APP_ID`

`frontend/index.html` 当前直接使用常量测试 `WX_APP_ID` 拼 OAuth 跳转地址，没有从后端读取配置。

### 问题 3：顾客页和商家页仍以内嵌测试值作为初始值

`frontend/customer.html` 与 `frontend/merchant.html` 虽然后续会请求 `/config/frontend`，但初始值仍然是测试 `WX_APP_ID`。这意味着即使接口异常、缺配置或返回空值，页面仍可能继续使用测试号。

### 问题 4：`/config/frontend` 缺少显式失败语义

`backend/main.py` 当前的 `/config/frontend` 直接返回：

- `app_id: settings.APP_ID`
- `base_url: ""`

当 `settings.APP_ID` 为空时，接口不会显式报错，前端也没有被强制阻断。

---

## 目标结构

本次改造后，配置流应变为：

1. 后端从环境变量读取微信配置，不再内置真实测试默认值
2. 前端在页面初始化时先请求 `/config/frontend`
3. 后端若发现 `WX_APP_ID` 缺失，则 `/config/frontend` 直接返回错误
4. 前端若拿不到有效 `app_id`，则停止后续 OAuth 跳转，并提示配置错误

这样可以保证：

- 配置来源唯一
- 错误暴露明确
- 已正确配置的环境不受影响

## 后端改造设计

### 一、调整 `WeChatSettings` 默认值策略

修改 `backend/wechat/config.py`：

- `WX_TOKEN` 改为空字符串默认值
- `WX_APP_ID` 改为空字符串默认值
- `WX_APP_SECRET` 改为空字符串默认值

即从“真实测试默认值”调整为“未配置即为空”。

本次不改动：

- `WX_ENCODING_AES_KEY`
- `WX_ADMIN_KEY`
- 支付配置字段

原因是本次范围仅针对用户明确指出的 3 项微信基础配置硬编码值。

### 二、为前端配置接口增加必填校验

修改 `backend/main.py` 中的 `GET /config/frontend`：

- 当 `settings.APP_ID` 为空或全空白时，直接返回 `503`
- 错误信息应明确说明 `WX_APP_ID` 未配置
- 当 `settings.APP_ID` 存在时，继续返回当前结构：
  - `app_id`
  - `base_url`

推荐行为：

- 状态码：`503 Service Unavailable`
- 错误消息：`WX_APP_ID 未配置`

选择 `503` 的原因是：

- 这是部署/配置层问题，不是前端参数错误
- 它比 `200 + 空字符串` 更容易暴露配置遗漏

### 三、保持其余后端逻辑不扩大改造

本次不新增统一 `validate_frontend_config()` 启动校验，也不扩大到所有 OAuth 相关接口统一失败控制。原因是本次目标是用最小改动去掉硬编码测试值，并先确保前端入口配置来源收敛。

如果后续需要进一步加强，可再单独补充“启动期校验微信登录配置”的改造。

---

## 前端改造设计

### 一、首页改为先拉取配置再发起 OAuth

修改 `frontend/index.html`：

- 删除写死测试 `WX_APP_ID`
- 在发起 OAuth 跳转前先请求 `/config/frontend`
- 从响应中读取 `app_id`
- 若接口失败或 `app_id` 为空，则提示配置错误并停止跳转

目标行为：

1. 页面加载
2. 请求 `/config/frontend`
3. 成功则拿到 `app_id`
4. 按现有方式拼接 OAuth 地址并跳转
5. 失败则提示“微信配置缺失”之类的显式错误

### 二、顾客页移除测试 `WX_APP_ID` 初始值

修改 `frontend/customer.html`：

- 删除测试 `WX_APP_ID` 初始值
- 保留 `initConfig()` 逻辑，但改为只有成功拿到 `cfg.app_id` 才允许后续 OAuth 跳转
- 若配置接口失败或 `cfg.app_id` 为空，则给出明确提示，不能继续隐式使用测试号

### 三、商家页移除测试 `WX_APP_ID` 初始值

修改 `frontend/merchant.html`：

- 删除测试 `WX_APP_ID` 初始值
- 保持与顾客页一致的配置获取和失败处理方式

### 四、前端错误处理要求

三页统一遵循：

1. 不再保留测试 `WX_APP_ID` 兜底
2. 配置接口失败时，显示用户可见提示
3. 配置缺失时，不继续构造 OAuth 跳转地址

本次不要求统一封装公共前端配置模块，避免为 3 个静态 HTML 页面引入额外抽象。

---

## 接口行为设计

### `GET /config/frontend`

成功响应保持兼容：

```json
{
  "app_id": "wx1234567890",
  "base_url": ""
}
```

缺少 `WX_APP_ID` 时返回：

```json
{
  "detail": "WX_APP_ID 未配置"
}
```

并使用 HTTP `503`。

这样前端无需猜测空值含义，可直接按错误路径处理。

---

## 测试与验证设计

### 一、后端自动化测试

新增或补充 `/config/frontend` 测试，覆盖至少两类场景：

1. 已配置 `WX_APP_ID`
   - 请求成功
   - 返回 `200`
   - `app_id` 为当前配置值

2. 未配置 `WX_APP_ID`
   - 请求失败
   - 返回 `503`
   - 错误消息明确为缺少配置

测试应尽量复用现有 FastAPI + httpx 异步测试框架，不额外引入新依赖。

### 二、前端手工验证

至少验证以下场景：

1. 设置 `WX_APP_ID` 后访问 `/`
   - 能正常发起 OAuth 跳转
2. 设置 `WX_APP_ID` 后访问 `/customer`
   - 页面初始化正常
   - 需要 OAuth 时可正常跳转
3. 设置 `WX_APP_ID` 后访问 `/merchant`
   - 页面初始化正常
   - 需要 OAuth 时可正常跳转
4. 不设置 `WX_APP_ID` 后访问上述页面
   - 不再使用测试号
   - 页面提示配置缺失或配置获取失败
   - 不继续 OAuth 跳转

## 风险与兼容性

### 风险 1：未配置环境会从“隐式可用”变为“显式报错”

这是本次设计的有意行为。它会暴露此前被测试默认值掩盖的问题，但这是为了避免上线时继续依赖错误配置。

### 风险 2：首页接入配置请求后，首次跳转依赖后端接口可用

这是合理约束，因为首页 OAuth 本来就依赖后端配置。只要部署环境配置正确，此行为不会影响正常使用。

### 风险 3：顾客页/商家页在配置异常时不再继续兜底

这同样是预期变化，目的是确保页面不会再误用仓库里的测试号。

- `frontend/index.html`

不新增复杂目录结构，不改动支付主链路。
